WAI-ARIA and WCAG Guideline 4.1

Auhor and date: Christophe Strobbe, 03.02.2016.

Background

In January 2016, members of the WCAG Working Group started a discussion on how errors related to the use of WAI-ARIA relate to the two success criteria in WCAG 2.0's Guideline 4.1. More specifically, the question was whether any ARIA-related errors could be considered as violations of WCAG Success Criterion (SC) 4.1.1. This document responds to that discussion.

WCAG 2.0 Guideline 4

SC 4.1 says the following

4.1.1 Parsing: In content implemented using markup languages, elements have complete start and end tags, elements are nested according to their specifications, elements do not contain duplicate attributes, and any IDs are unique, except where the specifications allow these features. (Level A)

Note: Start and end tags that are missing a critical character in their formation, such as a closing angle bracket or a mismatched attribute value quotation mark are not complete.

(Copyright W3C.)

The success criterion was written in order to protect assistive technologies (AT), especially screen readers, against markup errors that would require automatic repair functionality. Assistive technologies rely on the data exposed through the browser's accessibility API but perform additional parsing on top of that. This additional burden should not be made heavier by errors that are fairly easy to fix in the markup (i.e. by web site authors) but much harder to fix automatically (i.e. by AT).

The SC was written HTML 4.01 (published in December 1999) was still the most up-to-date non-XML version of HTML, and XHTML 1.0 (Second Edition, August 2002) and XHTML 1.1 - Module-based XHTML (May 2001) were the most up-to-date XML-based versions of HTML, the W3C's Markup Activity was still working on XHMTL 2.0, and the Protocols and Formats Working Group was still working on WAI-ARIA. SVG and SMIL, both XML-based, had been around for a while. (The first working draft of HTML5 had not yet been published when WCAG published it's second Last Call Working Draft.)

WCAG 1.0 checkpoint 3.2 required valid markup, but this requirement seemed too strict. Many people felt that requiring valid markup would draw too many resources towards fixing markup errors that had no impact on accessibility. (Sidenote: Back in 2005, not all W3C specifications had a conformance section that required valid markup.)

Requiring well-formed markup was considered as an alternative requirement but was rejected because well-formed markup is a concept that only applies to XML-based content. Well-formed markup has no meaning in the context of SGML-based languages such as HTML 4.01 and its predecessors. It would later also become meaningless to HTML5, which is not based on SGML or XML.

For these reasons, the WCAG working group identified four requirements that would apply to any web content that uses elements and attributes. The next few paragraphs will discuss these four requirements in more detail.

WAI-ARIA and WCAG 2.0 Guideline 4.1

WAI-ARIA is not a markup language like HTML or SVG. You cannot implement web content using only ARIA, so WCAG SC 4.1.1 does not apply to ARIA directly. However, WCAG SC 4.1.1 does not identify requirements for web standards; it identifies requiremnts for web content. WAI-ARIA is always used in web content that is implemented using other web standards (HTML, SVG, …). All of WCAG applies to content that uses HTML + WAI-ARIA, even when that content is generated by means of client-side scripts.
